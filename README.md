# Part 1. 자동화 도구 비교 구현 보고서 (n8n vs Make)

## 1. 워크플로우 개요

**주제**: 시험 채점 + 합격 여부 기록 및 통보 자동화

**흐름 요약**

Google Form 제출(자체 채점) → 점수 3단계 조건 분기
→ Google Sheets 명단 기록 + Discord 알림
→ 불합격자는 7일 뒤 재시험 안내 예약 발송


| 구성 요소 | 내용 |
|---|---|
| Trigger | Google Sheets 응답 시트에 새 행 추가 감지 |
| 조건 분기 | 정답 개수 기준 — 정회원(10개) / 준회원(6\~9개) / 불합격(0\~5개) |
| Action | 명단 시트 기록 → Discord 알림 → (불합격자만) Wait/Sleep 7일 후 재시험 안내 |

- 채점은 Google Forms 자체 퀴즈 기능으로 처리, 자동화 도구는 그 점수를 가져와 분기·기록·알림만 담당
- 분기 조건: `100점=정회원` / `60점 이상=준회원` / `60점 미만=불합격`

## 2. 도구별 구현 요약

### 🔵 n8n
- Google Sheets Trigger → **Code 노드**로 빈 값(0점)을 JS로 직접 보정 → **Switch**로 3단계 분기 → 각 branch에서 Sheets 기록 + Discord(HTTP Request) 알림, 불합격자는 Wait 노드 후 재안내 발송
- 어려웠던 점: 0점이 빈 값으로 들어오는 이슈, Code 노드 실행 모드(All Items/Each Item) 혼동, HTTP 요청 이후 원본 데이터 유실(`$('노드이름')`으로 재참조 필요), Discord 개인 DM 불가(채널 알림으로 대체)

### 🟠 Make
- Google Sheets **Watch Rows** → **Set Variable**에서 "100 / 100" 텍스트 점수를 `substring`/`indexOf`/`ifempty`로 숫자만 추출 → **Router + Filter**로 3단계 분기 → Sheets 기록 + HTTP(Discord) 알림, 불합격자는 **Sleep** 후 재안내 발송
- 어려웠던 점: 점수 텍스트/빈값 파싱, Router는 경로별 조건이 독립 검사되므로 범위를 겹치지 않게 직접 지정해야 함, 날짜 로케일(오전/오후 vs am/pm) 이슈, 시트에 시간 입력 시 자동 시리얼 넘버 변환 문제

## 3. 비교 요약

| 항목 | n8n | Make |
|---|---|---|
| UI/UX | 노드 플로우, Code 노드로 자유롭게 JS 작성 | 원형 모듈 연결, Router 분기가 시각적으로 직관적 |
| 난이도 | 코드 지식 필요하지만 유연 | 코드 없이 함수 조합만으로 접근 가능, 단 Make 특유 문법 필요 |
| 무료 플랜 | Wait 사용 시 오퍼레이션 소모 없음 → 장시간 대기에 유리 | 월간 오퍼레이션 제한 있어 대기 로직에 제약 |
| 디버깅 | 노드별 입출력 JSON 확인 쉬움, 에러 위치 파악 용이 | History에서 성공/실패 확인 가능하나 에러 메시지가 축약적 |
| 분기 방식 | Switch — 순서대로 첫 매칭만 실행 | Router — 각 경로 독립 검사(조건 겹침 주의) |
| 데이터 처리 | 비정형 데이터를 JS로 유연하게 보정 | 텍스트 파싱에 함수 중첩 필요, 식이 길어짐 |

### 장단점
- **n8n**: Code 노드로 세밀한 예외 처리 가능(✅) / HTTP 이후 원본 데이터 참조 문법이 진입장벽(❌)
- **Make**: 시각적 분기로 구조 파악 쉬움(✅) / 반복되는 데이터 타입 이슈 해결에 함수 중첩 필요(❌)

### 상황별 추천
- 빠른 간단 자동화·비개발자 → **Make**
- 복잡한 조건 분기·세밀한 데이터 제어·무료 운영 최적화 → **n8n**

## 4. 최종 결론

같은 워크플로우를 두 도구로 구현해보니, 차이는 "무엇을 만들 수 있는가"가 아니라 "문제(비정형 데이터)를 어떻게 해결하는가"에서 드러났다. n8n은 Code 노드로 즉시 보정 가능한 반면, Make는 함수 중첩이 필요했다. 반대로 구조 파악과 초기 구축 속도는 Router가 시각적으로 명확한 Make가 앞섰다.

두 도구의 공통점은 **HTTP 요청 이후 원본 데이터가 아닌 응답값이 다음 단계로 넘어간다**는 것이었고, 결국 핵심 역량은 특정 도구 사용법이 아니라 "데이터가 각 단계를 지나며 어떻게 변형되는가"를 추적하는 사고력이라는 걸 체감했다.

**도구 선택 기준**
1. 데이터가 지저분하거나 예외 처리가 많다 → n8n
2. 비개발자와 함께 유지보수해야 한다 → Make
3. 장시간 대기·무료 운영 부담 → n8n
4. 검증된 SaaS 간 단순 연동을 빠르게 → Make

이번 과제처럼 "조건 분기 + 비정형 데이터 + 시간 지연 발송"이 섞인 경우엔 최종적으로 **n8n**을 선택하겠다. 다만 Make로 프로토타입을 빠르게 검증한 뒤 n8n으로 정식 구현하는 하이브리드 접근도 유효하다고 본다.

# Part 2. 자유 주제 자동화 설계 및 구현 보고서 (n8n)

**주제**: 시험 채점 + 합격 여부 기록 및 통보 자동화 + **오답 AI 피드백**

## 1. 반복 업무 정의

구글 폼 제출 시 자동 채점 → 정회원/준회원/불합격 3단계 분류 → 스프레드시트 기록 → Discord로 결과 통보 → **준회원·불합격자에게는 AI가 오답 피드백 생성해 함께 발송** → 불합격자는 7일 후 재시험 안내 자동 발송

## 2. 도구 선정: n8n

- Wait 노드는 오퍼레이션 소모 없이 장시간 대기 가능 (7일 재시험 발송에 유리)
- Set 노드 표현식으로 빈 값(0점) 등 비정형 데이터 보정 용이
- Switch 노드가 순서대로 검사해 3단계 분기를 명확히 제어

**제외한 도구 (Make)**: 점수가 "100 / 100" 텍스트로 들어오는 등 타입 이슈가 반복돼 함수 중첩이 많아지고 유지보수가 복잡해질 우려

## 3. 워크플로우

[Google Forms 응답 수신] → [Set으로 정규화] → [Switch 점수 분기]
→ 정회원(100점) / 준회원(60~99점) / 불합격(60점 미만)
→ [시트 저장] + [AI 피드백(준회원·불합격만)] + [Discord 알림]
→ 불합격자는 [Wait 7일] → [재시험 안내 알림]


## 4. 구현 및 결과

- 세 케이스(정회원/준회원/불합격) 모두 시트 기록 및 Discord 알림 정상 동작 확인
- 준회원·불합격자에게는 OpenAI(GPT-4o mini)가 생성한 개인별 오답 피드백이 알림에 포함되어 발송됨
- 불합격자는 Wait 경과 후 재시험 안내 알림까지 정상 수신
- Executions 탭에서 전체 실행 이력 확인 가능

## 5. 특이사항 / 어려웠던 점

- OpenAI 노드 크레딧 소진 방지를 위해 준회원·불합격 분기에서만 AI 호출되도록 설계
- Forms 빈 응답이 null로 들어와 Set 노드에서 0점 처리 로직 추가
- Switch 경계값(60점, 100점) 설정 시 중복 분기 주의 필요
- Wait 7일 테스트가 어려워 1분으로 축소 테스트 후 7일로 재설정

## 6. AI 연동

- **모델**: OpenAI GPT-4o mini
- **역할**: 준회원·불합격자의 틀린 문제를 분석해 개인별 피드백 문구 자동 생성
- **비용 관리**: Switch에서 해당 분기로 넘어온 경우에만 OpenAI 노드 호출
